## 二、逐章评审

### 2.1 第一章：Claudian 断连诊断

| 项 | 方案原文 | 评审意见 |
|---|---|---|
| 根因 | DeepSeek 不再接受 `opus[1m]` 别名 | ✅ 逻辑合理，时间线吻合（6/20通→6/21不通），但缺少 DeepSeek 官方公告或错误日志佐证。建议补充：检查 Claudian 的 console 日志，确认返回的 HTTP 状态码和错误消息 |
| 修复方案一 | 改模型为 `claude-opus-4-8` | ⚠ **需验证**：DeepSeek 的 Anthropic 兼容端点是否真的接受 `claude-opus-4-8` 这个完整模型ID。如果 DeepSeek 本身只代理 `claude-sonnet-4` 等有限模型，改ID也可能不通。建议先在终端curl测试：`curl https://api.deepseek.com/anthropic/v1/messages -H "x-api-key: YOUR_KEY" -H "content-type: application/json" -H "anthropic-version: 2023-06-01" -d '{"model":"claude-opus-4-8","max_tokens":10,"messages":[{"role":"user","content":"hi"}]}'` |
| 修复方案二 | 改为 sonnet | ✅ 合理的降级备选 |
| 修复方案三 | 直接用 Anthropic API | ✅ 最稳但需额外付费，适合对稳定性要求高的场景 |
| 紧急度 | 🔴 1分钟修复 | ⚠ **过于乐观**。模型ID替换本身1分钟，但验证连通性、测试响应质量至少需要10-15分钟。建议标注为"15分钟含验证" |

**额外发现**：实勘 `realclaudian/data.json`，当前只有 tab 状态数据（`tabManagerState`），**看不到 provider 配置或模型名称的持久化存储**。这意味着模型配置可能存在 Claudian 的内部存储机制中而非 data.json。方案建议"直接在 claudian-settings.json 的 savedProviderModel.claude 字段修改"，但该文件在知识库目录内不存在。**修改路径需重新确认**，可能要通过 Obsidian UI 操作而非直接编辑文件。

### 2.2 第二章：知识管理方法论体系

| 项 | 评审意见 |
|---|---|
| 三层架构（捕获→组织→产出） | ✅ 经典且适用，与你的 PARA 变体吻合 |
| 四大方法论 | ✅ 选取合理。PARA+Zettelkasten+MOC+IMR 覆盖了知识管理的核心场景 |
| 09-项目资料分析 | ⚠ 方案说"20+项目"，实勘发现**有96个子目录**（含子目录），远超预估。说明该目录的整理需求比方案描述的更迫切 |
| 结构评价 | 基本准确，但有两处调整建议（见下方） |

**评审修正**：

1. `02-业务中台` 和 `02-工程图纸` 共用编号02，方案建议合并或独立项目。**建议保留图纸独立**——图纸体量大、查询模式与业务中台不同，合并反而降低检索效率。但建议将 `02-工程图纸` 改为独立编号（如 `13-工程图纸`），避免编号冲突导致的 Dataview 分类歧义。

2. `03-合同与法务` 和 `03-标准模板` 同样有编号冲突。**建议将标准模板并入03下的子目录而非平级**，或标准模板独立为 `14-标准模板`。

3. **缺省盲区**：方案未提及 `12-项目案例库` 与 `09-项目资料` 的边界划分。实勘中11-材料设备价格库已有实质内容（Excel+MD），而12-项目案例库是空目录还是已有案例？这决定了是否需要建立案例→项目的反向链接体系。

**Local REST API 修正**：该插件已运行在 27124(HTTPS) 和 27123(HTTP)，API Key 已生成。方案中的"启用 Local REST API 插件"这一行动项应改为"确认 Local REST API 可用性并配置 Python 脚本调用"。

#### Dataview 查询语法修正

方案原文三个查询均有问题：

```dataview
// ① 原文：WHERE 状态 != "归档"
// 问题：Dataview 中 != 应写作不带引号的比较
// 修正：
TABLE 项目名称, 合同金额, 日期, 状态
FROM "09-项目资料"
WHERE status != "归档"   // 注意：中文frontmatter字段需用英文key，或用 this.状态
SORT 日期 DESC

// ② 原文：WHERE 材料名称 = this.材料名称
// 问题：this.材料名称 只在笔记上下文中有效，单独运行无意义
// 修正：应改为具体查询或去掉WHERE做全表浏览

// ③ 原文：TASK FROM "00-工作台" AND "01-造价核心"
// 问题：Dataview TASK 查询不支持 AND 连接多个路径
// 修正：
TASK FROM "00-工作台" OR "01-造价核心"
WHERE !completed
```

#### Templater 模板修正

方案建议的造价笔记模板使用 `<%tp.date.now("YYYY-MM-DD")%>` 语法，但你**现有的模板使用 `{{date}}` 语法**——这是Obsidian核心模板语法而非Templater语法。你的 Templater 插件已启用，应统一使用 Templater 的 `<%tp%>` 语法以获得变量注入能力。

**建议**：将现有3个模板升级为 Templater 语法，而非从零新建。

## 三、方案缺失（盲区补充）

### 盲区1：知识库安全与备份

21,527个文件的知识库，方案未提及：
- 自动备份策略（Obsidian Git / iCloud / 定期ZIP）
- 误操作回滚机制
- Local REST API 开放后的访问控制（当前 API Key 在 data.json 明文存储）

**建议**：最低限度配置 Obsidian Git 自动提交，每天一次。

### 盲区2：数据质量与元数据标准化

09-项目资料 有96个子目录，但方案未检查这些目录下的 frontmatter 一致性。如果项目笔记的元数据字段不统一（有的写`合同金额`，有的写`合同价`，有的没有元数据），Dataview 查询会大面积漏数据。

**建议**：在行动清单中增加"元数据标准化"步骤，先统一 frontmatter 模板，再建 Dataview 查询。
