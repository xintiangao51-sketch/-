# 内观·造价工作台日结模板

> 灵台"内观三段式"改造吸收版，用于造价工程师日终复盘。
> 黄川灵魂镜像启发：需求洞察 → 工作日报 → 第一人称自传。
> 来源：灵台项目 `台律/运行手册.md §4.7`。

---

## 何时使用

| 触发 | 方式 | 输出 |
|:----|:----|:----|
| **显式** | 用户说"今日总结"/"日结"/"收工" | 老高生成三段式 |
| **兜底** | 每晚 22:00 自动化 | 统计型摘要（无 LLM 主观判断）|
| **手动** | 复制本模板填写 | 当日复盘 |

---

## 三段式结构

### 第一段：需求洞察

> 昕田今天真正在找什么？有什么没说出来但在行为中流露的需求？

**示例提问**（老高自我反思用）：
- 今天对话中，哪些问题用户反复追问？
- 用户今天表达过的 3 个痛点是什么？
- 用户今天最关心的金额/费率/风险点是哪一类？
- 用户有没有暗示想要什么但没明说（如"你有没有 X 的资料"= 用户在意 X）

**模板填写**：

```markdown
## 需求洞察
- 显式需求：
- 隐式需求：
- 未达成的需求（明日补足）：
```

---

### 第二段：工作日报

> 今天完成的关键操作和产出（基于事实，不靠 LLM 记忆）。

**数据来源优先级**：

```bash
# 1. git log（事实第一）
git log --since="today" --format="%h %s"

# 2. 知识库总索引变化（新增页数）
diff <(git show HEAD~1:知识库总索引.md) 知识库总索引.md | grep "^+"

# 3. 待提炼原料数
grep -l "^处理状态: 待提炼" 09-项目资料/**/*.md | wc -l
```

**模板填写**：

```markdown
## 工作日报
- 完成的关键操作（git commit 摘要）：
- 新增/更新知识页：
- 待提炼原料：X 篇
- 处理的签证/合同/询价：
- 数据更新（信息价/费率/定额）：
```

---

### 第三段：第一人称自传

> AI 以"我"的口吻回顾今天，检验理解是否偏差（理解偏了一看就知道）。

**核心要求**：
- 第一人称（"我"），克制简短
- 不要堆砌细节，1-3 句话
- 如有理解偏差，老高主动指出并提纠正方案

**模板填写**：

```markdown
## 第一人称自传
[例：今天我帮昕田处理了哈密国源 03 号签证的费率争议。我注意到他反复追问"管理费基数含不含规费"，说明他对此处有不确定。我应该把"管理费基数"作为单独知识卡片沉淀，避免下次重复。今天没出错，但我下次应该更主动地提示"原文未提及"的边界。]
```

## 完整填写示例

```markdown
# 内观 · 2026-06-25

## 老高 视角

### 需求洞察
- 显式需求：哈密国源 1.93 万㎡综合服务中心 03 号签证费率争议
- 隐式需求：管理费基数是否含规费（反复追问 3 次）
- 未达成：未提供 09-项目资料/哈密国源/合同/ 的费率条款页码

### 工作日报
- 完成：变更评估三步法+三级确认制+5 个手册（B1-B4）落地
- 新增页面：变更评估.md、知识库运行手册.md、知识库结构.md、系统架构.md
- 提炼：灵台 20 文件 → 11 文件吸收合并
- 脚本：lint_xinjiang.py 跑通，扫描 870 文件识别 133 缺口
- 待提炼原料：3 篇（青河 G331 结算争议文档）

### 第一人称自传
今天我帮昕田把灵台项目的 4 个核心机制（三级确认制/写回链路/枢纽页+辐页/台律迭代）反哺进五件套。我注意到他对"半结构化吸收 vs 全盘克隆"的决策很果断——直接选 B 档方案。我下次应该更早提示"C 档提炼吸收"的边界，避免过度技术细节。
```

## 与现有体系的对位

| 老高机制 | 本模板 |
|---------|--------|
| 工作日报 | 来自 git log（事实），不靠 LLM 记忆 |
| 需求洞察 | LLM 总结会话上下文 |
| 第一人称自传 | 检验理解偏差，1-3 句话 |
| 22:00 兜底 | 自动化生成统计型摘要 |

---

## 注意事项

- **不要省略三段**——需求洞察+工作日报+第一人称自传缺一不可
- **第一人称必须真实**——发现理解偏差就立刻指出，不要粉饰
- **不写流水账**——工作日报只列关键 3-5 项，不写所有细节
- **数据优先**——工作日报能用 git 命令取数据就不要靠 LLM 记忆
