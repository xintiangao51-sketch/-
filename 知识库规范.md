﻿# 知识库规范

鼎梁筑 AI 工程协作知识库的"宪法"——规定文件命名、内容位置、读写权限。

## 一、文件命名规范

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| 目录 | `编号-中文名` | `01-造价核心`、`10-索引与导航` |
| 文档 | 中文描述性名称 | `新疆市政定额2024说明.md` |
| 模板 | `模板-用途` | `模板-变更签证单.xlsx` |
| 项目 | `项目-名称` | `项目-乌鲁木齐道路改造/` |
| 归档 | 原名+日期 | `新疆市政定额2024说明_20260622归档.md` |

## 二、目录结构规范

```
D:\知识库\
├── 🏠 知识库首页.md       # 唯一首页（Obsidian Home）
├── 知识库规范.md          # 规范定义（本文件）
├── 知识库索引.md          # 已废止跳转页 → 首页
├── inbox/                # 知识库级半成品
├── outputs/              # 知识库级已确认交付物
├── 00-工作台/            # 造价工作台
│   ├── inbox/            # 工作台级半成品
│   └── outputs/          # 工作台级已确认交付物
├── 01-造价核心/          # 核心知识
├── ...                   # 其他分类目录
└── 99-归档库/            # 过期内容
```

## 三、inbox/outputs 边界规则

这是防止记忆污染的关键边界。

| 目录 | 用途 | 写入条件 | 读取权限 |
|------|------|----------|----------|
| `inbox/` | 半成品、草稿、研究材料 | 任何Agent可写 | 任何Agent可读 |
| `outputs/` | 已确认可交付内容 | 仅人工确认后写入 | 任何Agent可读 |

**铁律**：
- inbox → outputs 必须经过人工确认
- 未确认的内容不得进入 outputs
- outputs 中的内容可直接用于交付

## 四、三级证据链存储

| 级别 | 内容 | 存放位置 | 信任等级 |
|------|------|----------|----------|
| 一级证据 | 原始资料（论文、网页快照、定额原文） | `raw/` 或原目录 | A级，可直接引用 |
| 二级证据 | 提炼的事实和数据（标注来源） | `inbox/` | B级，需标注来源 |
| 三级结论 | 经交叉验证的结论 | `outputs/` 或 `pages/` | 需双重验证 |

## 十一、L1/L2/L3 分级确认制

消除 AI "过问"问题的核心机制——你的负担仅审查 L2.5 以上。

| 等级 | 行为 | 确认 | 适用场景 |
|------|------|:----:|---------|
| L1 自动放行 | AI 直接执行 | ❌ | 纯事实/数据更新、用户已有文章片段、已知框架扩展 |
| L2 摘要通知 | AI 执行后写摘要 | ❌ | 已知域新角度（≤3 篇原料浇灌同一页） |
| L2.5 简要确认 | 3行摘要过目 | ⚠️ | ≥5 篇原料批量灌同页 |
| L3 新域建页 | 建新页+write | ✅ | 全新概念/新框架引入 |
| 🚨 知识库规则变更 | 必须确认 | ✅ | 改目录结构/改规范/改命名规则 |

**核心原则**：Git 保底可回滚 → L1/L2 大胆放行 → 出问题 git revert。

### 等级判断指南

**L1 自动放行（无需确认）**：
- 纯事实更新：如日期、数字、名称等明确信息
- 用户已有文章片段：直接引用用户提供的原文
- 已知框架扩展：在已有页面添加同类内容
- 格式修正：错别字、格式调整等

**L2 摘要通知（执行后通知）**：
- 已知域新角度：用 ≤3 篇原料提炼同一页面
- 常规知识更新：不影响核心框架的信息更新
- 小幅内容调整：不影响整体结构

**L2.5 简要确认（需你过目）**：
- ≥5 篇原料批量灌入同一页面
- 涉及多个子页面的内容整合
- 可能影响原有结构的调整

**L3 新域建页（需你确认）**：
- 全新概念引入
- 新框架建立
- 重要域的首次内容

**🚨 知识库规则变更（必须确认）**：
- 目录结构变更
- 规范文件修改
- 命名规则变更

### 你的职责

- 仅审查 L2.5 以上等级
- L1/L2 完全放行 AI 自动执行
- 信任 Git 可回滚机制
- 定期回顾整体质量

## 十二、提炼后检查清单（每轮必检）

借鉴灵台提炼后流程 + 适配 Agent 工作流：每轮提炼完成后必执行的7步收尾流程。

> **注意**：此流程与 inbox→outputs 规则不冲突。提炼产物直接进 outputs 区（你的 A 级目录），而 inbox 区保持半成品状态。

### 检查清单流程

| 步骤 | 检查内容 | 执行 Agent |
|:----:|----------|----------|
| 1 | 交叉链接：新页底部推荐阅读 3-6 条 + 关系说明 | Claude/Codex |
| 2 | 来源回链：每篇原料末尾加 → [[丹房/目标页]] | Codex |
| 3 | 更新索引：索引.md 追加新行（每页一行摘要） | Codex |
| 4 | ⚡漂移检查：新页是否"绕不过"→加标；已有⚡是否退化→移标 | Claude |
| 5 | 日志追加：写入 日志.md | Codex |
| 6 | Git 提交：先写日志→校验→提交 | Codex |
| 7 | 内观判定：本次会话结束？→ 生成内观 | Hermes/WB |

### 步骤详解

**步骤1：交叉链接**
- 在新页底部添加"推荐阅读"部分
- 推荐 3-6 条相关内容
- 使用 [[双链]] 格式
- 示例：
  `
  ## 推荐阅读
  - [[相关页面1]] — 关联说明
  - [[相关页面2]] — 关联说明
  `

**步骤2：来源回链**
- 在每篇原料文件末尾添加来源回链
- 标注提炼后的目标页面
- 示例：
  `
  ---
  → 已提炼至：[[01-造价核心/定额与计价/具体页面]]
  `

**步骤3：更新索引**
- 在 10-索引与导航/索引.md 追加新行
- 格式：- [[页面路径]] — 摘要（≤15字）
- 示例：- [[01-造价核心/新页面]] — 新页面摘要

**步骤4：⚡漂移检查**
- 判断新页面是否应该标记 ⚡核心
- 判断已有页面是否需要移除 ⚡标记
- 遵循"绕不过"原则

**步骤5：日志追加**
- 写入操作日志
- 记录时间、操作、内容
- 示例：
  `
  ## 2026-06-26
  - 新增：定额与计价-新页面
  - 来源：X篇原料
  `

**步骤6：Git提交**
- 先写提交日志
- 校验变更内容
- 执行提交
- 示例提交信息：eat: 新增定额与计价核心知识卡片

**步骤7：内观判定**
- 判断本次会话是否结束
- 如需结束，生成内观总结
- 记录关键进展和下一步

### 与 inbox→outputs 的关系

| 流程 | 目标 | 说明 |
|------|------|------|
| inbox→outputs | 半成品→确认交付 | 人工确认后的交付物流程 |
| 提炼后检查清单 | 提炼产物→标准收尾 | AI 完成提炼后的7步收尾 |

**不冲突说明**：
- inbox→outputs：管控内容质量，确保交付物可靠性
- 提炼后检查清单：标准化收尾，确保可追溯和关联

两者互补：一个管质量，一个管流程。

## 十三、枢纽页+辐页架构（防长文堆积）

解决页面线性膨胀问题——单页过长时拆分为"枢纽页+辐页"结构。

### 拆页触发条件（满足任一即拆）

| 条件 | 说明 |
|------|------|
| 单页 > 100 行 | 拆出最独立的 2-3 个角度为辐页 |
| 单页 > 7 个 ## 节 | 合并相关节为辐页 |
| 单次提炼新增 > 5 篇原料灌同页 | 按主题分拆 |

### 辐页命名规范

- 格式：枢纽名·角度.md
- 示例：造价实战经验库·安装工程.md、造价实战经验库·市政道路.md

### 枢纽页规范

- **保留内容**：
  - 核心定义
  - 框架速览表
  - 各辐页入口
- **行数控制**：60-80 行
- **节数控制**：5-7 节以内
- **详细展开**：链到辐页

### 结构示例

`
造价实战经验库/
├── 造价实战经验库.md          # 枢纽页（~70行，6节）
├── 造价实战经验库·安装工程.md   # 辐页1
├── 造价实战经验库·市政道路.md   # 辐页2
└── 造价实战经验库·房建结构.md  # 辐页3
`

### 枢纽页模板

`markdown
# 造价实战经验库

> 核心定义一句话

## 📊 框架速览

| 分类 | 内容 |
|------|------|
| 辐页1 | [[辐页1标题]] |
| 辐页2 | [[辐页2标题]] |

## 🚀 快速入口

- [[辐页1标题]] — 简述
- [[辐页2标题]] — 简述

## 📖 详细内容

详见各辐页...
`

## 十四、写回链路（不丢好答案）

解决AI对话中产生的好答案丢失问题——直接写入知识库，不经过inbox。

### 核心规则

**写回链路**：对话中产生有通用价值的分析/对比/结论，直接写入知识库，不经过 inbox。

### 触发条件

| 类型 | 说明 | 示例 |
|------|------|------|
| 分析 | 深度分析报告 | 造价趋势分析、市场对比 |
| 对比 | 多维度比较 | 定额对比、方案比选 |
| 结论 | 关键结论 | 决策建议、经验总结 |
| 方法论 | 通用方法论 | 报价策略、成本拆解 |

### 写入位置

- 目录：对应域目录/outputs/
- 示例： 1-造价核心/outputs/、 5-投标报价/outputs/

### Frontmatter标记

`yaml
---
来源: 写回
类型: 分析/对比/结论/方法论
会话: YYYY-MM-DD
---

# 内容标题

> 核心结论一句话

## 分析/对比/结论正文

...
`

### 与提炼产物的区别

| 维度 | 写回 | 提炼 |
|------|------|------|
| 来源 | AI对话 | 原料提炼 |
| 品质 | 对话产出 | 原料加工 |
| Frontmatter | 来源: 写回 | 品级: A/B/C |
| 原料数量 | 无 | 有 |
| 位置 | outputs/ | outputs/ |

### 写回产物示例

`markdown
---
来源: 写回
类型: 对比
会话: 2026-06-26
tags: [对比, 报价策略]
---

# 常用定额体系对比分析

> 新疆2020版 vs 国标清单适用范围对比

## 适用场景对比

| 定额体系 | 适用场景 | 优劣势 |
|---------|---------|--------|
| 新疆2020版 | 新疆地区项目 | 地域适配强 |
| 国标清单 | 清单报价 | 全国通用 |

## 结论

- 清单报价优先用国标
- 成本核算优先用新疆2020版
...
`

### 写回流程

1. 对话中识别有通用价值的内容
2. 判断是否符合触发条件
3. 确定写入位置（对应域outputs/）
4. 添加Frontmatter标记
5. 写入文件并更新索引

## 十五、内观三段式（每日会话收尾）

Hermes接管内观生成——AI会话的每日复盘机制。

### 触发条件

| 触发类型 | 条件 | 说明 |
|----------|------|------|
| 显式触发 | 你说"结束"/"今天到这" | 立即生成 |
| 兜底触发 | 每日 22:00 | 检查内观.md是否有今日条目，无则生成统计型 |

### 写入位置

- 文件：D:\知识库\10-索引与导航\内观.md
- 格式：内观 · YYYY-MM-DD.md（可选）

### 内观三段式结构

`markdown
# 内观 · YYYY-MM-DD

## Hermes 视角

### 需求洞察
昕田今天真正在找什么？有什么没说但流露的需求？

### 工作日报
今天完成的关键操作和产出（基于 git log + 日志.md）

### 第一人称自传
以"我"（AI）口吻回顾今天，检验理解是否偏差
`

### 规则说明

1. **需求洞察**：从你的问题/反馈中识别深层需求
2. **工作日报**：基于git log和日志.md总结完成的关键操作
3. **第一人称自传**：以"我"口吻回顾当天，检验理解偏差

### 多AI场景

- 多个AI活跃时，各自追加 ## {AI标签} 视角
- 示例：
  `markdown
  ## Hermes 视角
  ...
  
  ## Codex 视角
  ...
  `
