## 〇、总纲

本文件定义鼎梁筑 AI 工程协作体系的记忆工程（Memory Engineering）方法——如何让经验不丢、不混、不烂，跨项目可复用，越用越准。

六维方法论：
Prompt Engineering  → 把需求说清
Context Engineering → 把上下文管住
Harness Engineering → 把规则定好
Loop Engineering    → 把结果做准
Memory Engineering  → 把经验沉淀 ← 本文件
Tool Engineering    → 把工具用尽

### 0.1 两大理论基石

| 理论 | 来源 | 核心贡献 |
|------|------|----------|
| MAGMA 多图记忆架构 | ACL 2026 (arXiv:2601.03236) | 四维正交图谱、双流读写、意图路由 |

|------|------|

## 一、MAGMA 四维记忆图谱

MAGMA 核心创新：将混沌的经验解构为四个独立运作的逻辑维度，每个记忆项同时在四个图谱中建立关联。

### 1.1 语义图谱（Semantic Graph）

| 属性 | 说明 |
|------|------|
| 结构 | 无向图，节点=记忆项，边=余弦相似度 |
| 核心功能 | 构建概念共鸣网络，连接相似事件与概念 |
| 造价场景 | C25混凝土关联C30混凝土、商混、材料单价、定额子目 |
| 检索方式 | 向量相似度 Top-K |

### 1.2 时间图谱（Temporal Graph）

| 属性 | 说明 |
|------|------|
| 结构 | 线性序列，不可变主干 |
| 核心功能 | 历史编年史，基于时间先后记录所有事件 |
| 造价场景 | 信息价按月归档 → 趋势分析 → 价格预测 |
| 检索方式 | 时间范围 + 先后顺序 |

所有记忆项必须带时间戳。时间图谱一旦写入，不删除不修改，只追加新版本。

### 1.3 因果图谱（Causal Graph）

| 属性 | 说明 |
|------|------|
| 结构 | 有向图，A→B 表示 A 导致 B |
| 核心功能 | 逻辑推演路径，专门解答为什么类问题 |
| 造价场景 | 设计变更 → 签证费用 → 结算调增 |
| 检索方式 | 路径遍历 + 依赖链分析 |

因果图谱由慢车道异步推理生成，不要求实时完整。

### 1.4 实体图谱（Entity Graph）

| 属性 | 说明 |
|------|------|
| 结构 | 节点辐射，中心=实体，周围=关联信息 |
| 核心功能 | 为项目、合同、人员建立专属锚点 |
| 造价场景 | 哈密国源项目锚点辐射：合同金额、中标日期、总包、清单路径、结算状态 |
| 检索方式 | 实体名称 → 辐射关联 |

### 1.5 四图谱协同工作流

```
用户查询
    ↓
意图路由（判断查询类型）
    ├─ 是什么 → 语义图谱
    ├─ 什么时候 → 时间图谱
    ├─ 为什么 → 因果图谱
    └─ 谁/哪个项目 → 实体图谱
    ↓
策略引导图谱遍历（heuristic beam search）
    ↓
结构化上下文生成（带时间戳+来源+置信度）
    ↓
返回给查询Agent
```

## 二、双流读写机制

### 2.1 快车道（Fast Path：突触级摄入）

| 属性 | 值 |
|------|-----|
| 延迟 | 秒级 |
| 操作 | 切片 → 向量索引 → 更新语义图 + 时间图 |
| 场景 | 日常交互中实时记忆写入 |
| 阻塞 | 零阻塞，不中断用户交互 |

### 2.2 慢车道（Slow Path：结构巩固）

| 属性 | 值 |
|------|-----|
| 延迟 | 分钟~小时级（后台异步） |
| 操作 | 推演因果连接 → 推演实体连接 → 更深关联推理 |
| 场景 | 会话空闲时、定时任务触发、手动触发 |
| 触发 | 沃巴迪主责调度，或 Hermes 空闲时触发 |

### 2.3 双流在四Agent中的分工

| Agent | 快车道职责 | 慢车道职责 |
|-------|-----------|-----------|
| 赫尔墨斯 | 实时任务状态写入L1 | 生成记忆草案，判断是否落L2 |
| 克劳德 | 判断结论+依据写入L1 | 因果推演，争议推理入因果图 |
| 科德克斯 | 执行结果+验证评分写入L1 | 数据归档，批量记忆结构化 |
| 沃巴迪 | 运营数据实时更新台账 | 主责慢车道调度：复盘、沉淀、迭代 |

## 三、意图感知检索

系统自动识别用户查询意图，在多维图谱中点亮最相关的逻辑路径。

### 3.1 查询类型路由

| 用户意图 | 路由图谱 | 示例 |
|----------|---------|------|
| 概念/相似性查询 | 语义图谱 | C30混凝土单价大概多少？ |
| 时间/序列查询 | 时间图谱 | 哈密项目上月结算进度？ |
| 因果/推理查询 | 因果图谱 | 为什么道班房结算超了？ |
| 实体/关联查询 | 实体图谱 | G331项目有哪些合同？ |
| 复合查询 | 多图谱融合 | 哈密国源C30单价跟上次比涨了多少？ |

### 3.2 检索优先级

精确匹配（实体名称、定额编号、合同编号）> 语义相似 > 时间临近 > 因果路径

### 3.3 AgentMessage 记忆检索标记

在 thought 字段中标注记忆检索意图：

**Thought**: [MEM:实体]查询哈密国源项目合同金额...
**Thought**: [MEM:因果]分析道班房结算超支原因...
**Thought**: [MEM:语义]查找C30混凝土相关资料...

## 四、L1/L2/L3 增强记忆分层

在现有三层架构基础上，融合 MAGMA 四维图谱。

### 4.1 三层总览

| 层级 | 位置 | 容量 | 存什么 | 图谱覆盖 | 维护频率 |
|------|------|------|--------|----------|----------|
| L1 热记忆 | Agent 自身 memory | ~5000字 | 核心参数、当前项目、强偏好 | 语义+时间 | 每轮更新 |
| L2 温记忆 | D:\知识库\10-索引与导航\MEMORY-AGENT.md | 不限 | 项目经验、坑点库、交接记录 | 四图谱 | 每周 |
| L2-运营 | D:\知识库\16-运营数据\ | 不限 | 台账、收支明细、复盘报告 | 实体+时间+因果 | 每次运营更新 |
| L3 冷记忆 | D:\知识库 各分类目录 | 5.6GB | 原始资料、规范、模板、定额库 | 语义+实体 | 按需 |

### 4.2 L1 热记忆铁律

- 不超过 5000 字，满了就下沉 L2
- 只存：高频偏好、当前活跃项目（≤3个）、强规则
- 不存：临时参数、未验证判断、情绪化表达

## 五、记忆写入铁律

### 5.1 写入条件

应当写入：
- 长期偏好、稳定规则、已验证配置
- 项目事实（合同金额、中标日期、关键参数）
- 反复出现的坑点（含根因+修复+预防）
- 可复用模板和流程（经过≥2次验证）
- 多Agent协作经验
- 运营数据（合同额、回款、支出、利润）
- 经确认的费率/价格参数（含来源+日期）

不写入：
- 未确认猜测、临时参数、过期价格
- 密钥账号密码、涉密原文
- 情绪化表达、未验证模型判断（C/D级）
- 未确认财务数据、未验证项目状态变更

### 5.2 写入流程

正式任务完成 → Hermes判断是否需要记忆草案 → 草稿生成（四图谱标注）→ 昕田确认 → 精炼一句话入L1、详细全文入L2、原始资料入L3、运营数据入L2-运营 → 慢车道后台巩固

### 5.3 12 Rules 记忆铁律映射

| 规则 | 记忆映射 | 说明 |
|------|----------|------|
| Rule 6 Token Budget | 记忆Token预算 | L1≤5000字，每次检索≤4K tokens |
| Rule 7 Surface Conflicts | 暴露记忆冲突 | 矛盾信息并排标注，不混合 |
| Rule 10 Checkpoints | 记忆检查点 | 每完成一个任务阶段写一次检查点 |
| Rule 12 Fail Loud | 错误记忆 | 错误和失败必须写入记忆用于复盘 |

### 5.4 写回链路（灵台吸收·2026-06-25）

> 来源：灵台 `台律/运行手册.md §4.5 写回链路`
> 定位：AI 对话中产生的通用价值结论，应直接沉淀为知识库页面，不进入原料层。

**为什么需要写回链路**：好答案不应该消失在聊天记录里。Karpathy 风格——AI 产出的洞察、对比、结论，如果具有通用价值，应直接进知识库，减少提炼层的噪音。

**触发条件**：
- 对话中产生任何有通用价值的分析、对比、结论、发现
- 不只是"独立洞察"——只要对将来类似场景有帮助，就可归档
- 触达阈值：单次对话 ≥ 3 条有价值判断，或单项判断 ≥ 500 字深度

**写回目标位置**（按内容性质分）：

| 内容类型 | 写回位置 | FM 标记 |
|---------|---------|---------|
| 签证争议处理 SOP | `01-造价核心/变更索赔/` | 来源: 写回 |
| 投标报价策略 | `05-投标报价/` | 来源: 写回 |
| 图纸提量清单 | `01-造价核心/工程量/` | 来源: 写回 |
| 合同条款审查发现 | `03-合同与法务/` | 来源: 写回 |
| 定额套用对比 | `01-造价核心/定额与计价/` | 来源: 写回 |
| AI 工作流优化 | `07-AI工具与技能/` | 来源: 写回 |
| 跨 Agent 协作经验 | `10-索引与导航/AI-Agent五件套/` | 来源: 写回 |

**FM 规则**（区别于提炼产物）：
- 加 `来源: 写回` 字段（唯一标识）
- **不加** `品级` 字段（属于提炼体系）
- **不加** `原料数量` 字段（写回无原料来源）
- 加 `日期` 字段（写回时间）

**执行流程**：

```
对话产生洞察 → LLM 判断通用价值 → 用户确认写回
                │
                ├── L1 简单事实 → 直接写回 + git commit
                ├── L2 分析对比 → 写回 + 老高补 3-6 条推荐阅读
                └── L3 新框架 → 用户确认后写回 + 触发 §5.1 记忆铁律
```

**与提炼的区别**：

| 维度 | 提炼 | 写回 |
|------|------|------|
| 来源 | 09-项目资料/ 原料 | 对话洞察 |
| 触发 | "提炼"指令 | 对话中自然产生 |
| FM | 品级 + 原料数量 | 来源: 写回 |
| 路径 | 09-项目资料 → 知识页 | 对话 → 知识页 |
| 频次 | 周期提炼 | 每对话可能 |

**造价场景示例**（实际写回记录）：

| 日期 | 写回内容 | 目标位置 |
|------|---------|---------|
| 2026-06-25 | 灵台 4 大机制吸收复盘 | 本文件（§5.4） |
| 待写回 | 哈密国源 03 号签证费率争议处理 SOP | `01-造价核心/变更索赔/` |

## 六、Supermemory 工具接入

### 6.1 工具定位

Supermemory（22K+ GitHub Stars）是开源记忆 API，作为鼎梁筑记忆体系的外部持久化引擎。

| 功能 | 说明 |
|------|------|
| 事实提取 | 自动从对话中提取结构化事实 |
| 用户画像 | 静态事实+动态上下文，50ms返回 |
| 混合检索 | RAG + 记忆统一查询 |
| 自动遗忘 | 临时事实过期、矛盾事实版本化 |

### 6.2 安装方式

```bash
# 一键安装（推荐）
curl -fsSL https://supermemory.ai/install | bash
# npx本地运行
npx supermemory local
# Python包
pip install supermemory-agent-mcp
# Claude Desktop MCP
npx -y install-mcp@latest https://mcp.supermemory.ai/mcp --client claude --oauth=yes
```

### 6.3 与鼎梁筑记忆体系映射

| Supermemory 概念 | 鼎梁筑对应 | 说明 |
|-----------------|-----------|------|
| container_tag | 项目名 | 每个项目一个container |
| static profile | L1热记忆 | 昕田长期偏好和规则 |
| dynamic profile | L2温记忆 | 当前活跃项目上下文 |
| memory extraction | AgentMessage thought | 从thought提取关键事实 |
| hybrid search | 四图谱融合检索 | 语义+实体混搜 |

### 6.4 Dory 备选工具

Dory（84% LongMemEval）是本地优先的图谱记忆方案：

```bash
pip install dory-memory
```

| 对比项 | Supermemory | Dory |
|--------|-------------|------|
| 部署方式 | 本地服务/云端 | 单SQLite文件 |
| 图谱类型 | RAG+Profile | 4类型记忆 |
| 联网要求 | 可选 | 完全离线 |
| 适用场景 | 多Agent共享记忆 | 单Agent本地记忆 |

### 7.4 记忆健康评分

| 维度 | 分值 | 说明 |
|:----|:----:|------|
| 准确率 | 25 | 记忆与事实一致，无编造 |
| 完整率 | 20 | 关键字段齐全，无空缺未标注 |
| 使用率 | 15 | 记忆被有效检索利用 |
| 时效性 | 15 | 过期信息及时清理 |
| 结构化 | 15 | 四图谱标注完整 |
| 一致性 | 10 | 无冲突、无重复 |

90分合格。低于90分需要记忆维护。
