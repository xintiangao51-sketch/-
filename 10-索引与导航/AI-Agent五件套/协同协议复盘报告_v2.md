# 多Agent协同协议复盘报告 v2.0

> 版本：v2.0 | 复盘日期：2026-06-24
> 原版：v1.0（2026-06-22）
> 变更：从三Agent升级为四Agent协同体系，加入Workbuddy（沃巴迪）

---

## 一、v1.0复盘结论

### 1.1 已验证可用的部分

| 维度 | 状态 | 说明 |
|------|------|------|
| 三Agent分工 | ✅ 已跑通 | Hermes调度/Claude Code判断/Codex执行，001/002任务验证闭环 |
| 任务卡机制 | ✅ 可用 | YAML任务卡格式清晰，状态机流转合理 |
| inbox/outputs边界 | ✅ 关键机制 | 半成品与已确认交付物分离，防止记忆污染 |
| 三级证据链 | ✅ 文档化 | 一级证据→二级证据→三级结论，流程已定义 |
| 质检流程 | ✅ 跑通 | Hermes质检→缺口识别→补齐→归档 |
| 记忆分层 | ✅ 已建立 | L1热/L2温/L3冷三层，Holographic外置记忆已激活 |

### 1.2 暴露的问题

| 问题 | 根因 | 影响 |
|------|------|------|
| 项目交付后无人管 | 三Agent体系覆盖"从接活到交活"，不覆盖"交活后的运营管理" | 项目上线后财务、迭代、数据沉淀缺系统支撑 |
| 知识沉淀断层 | 项目经验散落在各Agent记忆中，缺乏统一的"第二大脑"做结构化沉淀 | 后续项目无法高效复用历史经验 |
| 财务管理缺位 | 无Agent负责收支管理、成本台账、利润分析 | 经营数据不透明 |
| 多人协作缺角色 | 三Agent各自为战，缺乏"运营中台"角色 | 项目间数据不互通 |

### 1.3 核心判断

> 三Agent解决的是"把活干好"，四Agent要解决的是"把活干好之后怎么办"。

## 二、v2.0升级要点

### 2.1 新增Agent：Workbuddy（沃巴迪）

| 维度 | 定义 |
|------|------|
| 中文名 | 沃巴迪 |
| 英文名 | Workbuddy |
| 部门 | 运营部 |
| 职位 | 运营部总监 / 第二大脑 |
| 一句话定位 | 项目落地后的管家：管理、运维、财务、迭代、数据沉淀 |
| 核心价值 | 让每个项目的交付不是终点，而是下一个项目的起点 |

### 2.2 四Agent总览

| Agent | 中文名 | 部门 | 一句话 | 管什么 |
|-------|--------|------|--------|--------|
| Hermes | 赫尔墨斯 | 调度部 | 中枢神经 | 任务怎么分、状态怎么转、记忆怎么存 |
| Claude Code | 克劳德 | 研判部 | 大脑 | 问题怎么想、方向怎么定、争议怎么裁 |
| Codex | 科德克斯 | 执行部 | 双手 | 文件怎么做、数据怎么处理、成果怎么交付 |
| Workbuddy | 沃巴迪 | 运营部 | 第二大脑 | 项目怎么管、财务怎么算、经验怎么沉淀 |

### 2.3 协同流变化

**v1.0 流程（三Agent）：**
```
用户提需求 → Hermes拆解 → Claude Code判断 → Codex执行 → Hermes质检 → 交付
```

**v2.0 流程（四Agent）：**
```
用户提需求 → Hermes拆解 → Claude Code判断 → Codex执行 → Hermes质检 → 交付
    ↓                                                                        ↓
    └──────────────── 交付后 → Workbuddy接管 ────────────────────────────────┘
                         ├── 项目运营管理
                         ├── 财务收支台账
                         ├── 数据分析与复盘
                         ├── 经验沉淀与知识库补充
                         └── 迭代需求收集与规划
```

### 2.4 共享知识库协议

**所有Agent共享同一个知识库：** `D:\知识库`

| 规则 | 说明 |
|------|------|
| 读权限 | 四个Agent均可读取知识库全部内容 |
| 写inbox | 四个Agent均可写入inbox（半成品） |
| 写outputs | 仅人工确认后可写入outputs（已确认交付物） |
| 沉淀规则 | Workbuddy主责沉淀，其他Agent辅助 |
| 冲突处理 | 同一文件多人需要修改时，Hermes协调，避免覆盖 |

| 任务类型 | 主Agent | 协作Agent | Workbuddy角色 |

| 深度判断/方案设计 | Claude Code | Hermes | 提供历史数据支撑 |
| 文件/数据/代码执行 | Codex | Claude Code | — |

### 3.3 交接协议更新

| 交接方向 | 卡格式 | 说明 |
|----------|--------|------|
| Hermes → Claude Code | 判断任务卡 | 需要判断的问题，不是要执行的动作 |
| Hermes → Codex | 执行任务卡 | 要处理的文件、输出的成果、验证方式 |
| Hermes → Workbuddy | 运营任务卡 | 要管理的项目、要沉淀的内容、要分析的数据 |
| Claude Code → Codex | 方案交接卡 | 判断结论、风险边界、执行要求 |
| Codex → Hermes | 执行反馈卡 | 完成内容、验证结果、质量评分 |
| Workbuddy → Hermes | 运营反馈卡 | 项目状态、财务数据、沉淀成果 |
| Workbuddy → Claude Code | 数据请求卡 | 需要分析的数据、需要判断的问题 |
| 任意Agent → 知识库 | 沉淀卡 | 写入inbox，Hermes质检后转outputs |
