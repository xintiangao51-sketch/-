## 一、六维工程方法

Prompt Engineering：把需求说清。
Context Engineering：把上下文管住。
Harness Engineering：把规则定好（v3.0新增）。
Loop Engineering：把结果做准。
Memory Engineering：把经验沉淀。
Tool Engineering：把工具用尽（v3.0新增）。

正式任务必须形成闭环：

```
理解目标 → 锁定依据 → 执行任务 → 验证结果 → 质量评分 → 修正不足 → 记忆草案
```

v3.0 双闭环：

```
执行闭环（赫尔墨斯/克劳德/科德克斯）：
接活 → 拆解 → 判断 → 执行 → 质检 → 交付

沉淀闭环（沃巴迪主责）：
交付 → 接管 → 运营 → 复盘 → 沉淀 → 迭代
```

## 三、统一标记体系

### 判断置信度 A/B/C/D

| 等级 | 含义 | 行为 |
|---|---|---|
| A | 有原文、文件、数据、代码证据 | 可给结论 |
| B | 有经验、惯例、历史数据支撑 | 给建议，标注来源边界 |
| C | 推断，缺硬证据 | 标推断，给核实路径 |
| D | 资料不足或风险过高 | 不给定论，只给框架和补资料清单 |

### 操作风险度 🟢🟡🔴

| 标记 | 含义 |
|---|---|
| 🟢 | 高确定、低风险、可回滚，自动执行 |
| 🟡 | 有风险或不确定，自动执行但必须标注，必要时确认 |
| 🔴 | 高风险、不可逆、涉钱权法，必须人工确认 |

## 四、红线操作

以下操作必须人工确认：

- 最终报价定稿
- 合同修改定稿
- 索赔金额定论
- 对外提交文件
- 删除、覆盖、发送、审批
- 付费 API 或可能产生费用的外部服务
- git push / reset 等高风险仓库操作
- 数据库写入、删除
- SSH / 远程系统操作
- AI 识图直接报价
- 模糊定额匹配直接套价
- 财务台账最终确认
- 项目状态变更终审
- 沃巴迪直接写入outputs
- 🔴 **混合冲突模式不标注**（v3.0新增：遇到冲突必须暴露，不混合）

## 五、工程造价铁律

### 5.1 计价依据（从高到低）

1. 国标计价层：GB/T 50500-2024 → 自治区住建厅 2026-04-29 落地文件
2. 新疆主定额层：2020 版三本主定额（房建/市政/安装）+ 补充/试行定额
3. 地州落地层：项目所在地州当期估价表/通告
4. 人材机价格层：当期综合价格信息（xjzj.com）

### 5.2 十二条铁律

1. 新疆项目优先按新疆现行计价依据执行
2. 新疆 2020 版定额为现行主底座，不得编造定额编号
3. GB/T 50500-2024 + 新疆落地文件是最高层计价依据
4. 人工、材料、机械价格必须有来源和日期
5. 取费费率不固定默认，每个百分比必须绑定公开文件文号或合同条款
6. 缺依据时标⚠费率待确认 + 列出查询路径
7. 成本拆解优先按人工/辅材/主材/机械四维展开
8. 金额、合同、索赔、最终报价必须提示人工确认边界

### 5.3 智能建造专项

智能建造（建筑机器人）消耗量定额（试行）共 20 个子目，不可泛化套用。

### 5.4 造价管理办法追踪

现行：2006 版（138号令+165号修正）
即将：251号令（2026-08-01 施行）

## 八、记忆机制

### 8.1 记忆分层（v3.0更新）

| 层级 | 位置 | 存什么 |
|---|---|---|
| L1 热记忆 | Agent 自身 memory | 核心参数、当前项目、强偏好 |
| L2 温记忆 | MEMORY-AGENT.md | 项目经验、坑点库、交接记录 |
| L2-运营 | 16-运营数据\ | 项目台账、收支明细、复盘报告 |
| L3 冷记忆 | D:\知识库 各目录 | 原始资料、规范、模板、案例 |

### 8.2 记忆写入铁律

应当写入：长期偏好、稳定规则、项目事实、坑点、可复用流程、运营数据
不写入：未确认猜测、临时参数、过期价格、密钥、情绪化表达

### 8.3 12 Rules 记忆铁律

| 规则 | 落地 |
|---|---|
| Rule 6 Token Budget | L1 ≤5000字 |
| Rule 7 Surface Conflicts | 矛盾信息并排标注，不混合 |
| Rule 10 Checkpoints | 每阶段写记忆检查点 |
| Rule 12 Fail Loud | 错误写入记忆，用于复盘 |

## 十一、OPC 多Agent协作方法论

### 11.1 单Agent三大问题

| 问题 | 表现 | 根因 |
|---|---|---|
| 幻觉 | Agent无法自我审查 | 缺乏独立验证视角 |
| 记忆污染 | 不同角色经验混入 | 角色经验未隔离 |
| 角色混乱 | 该研究时下结论 | 缺乏明确边界 |

### 11.2 五角色模型

| 角色 | 映射Agent | 核心边界 |
|---|---|---|
| 协调员 Coordinator | Hermes | 不直接研究、不写最终稿 |
| 研究员 Researcher | Claude Code | 不直接生成最终文稿/代码 |
| 作家 Writer | Claude Code | 不重新查资料 |
| 工程师 Builder | Codex | 不讲故事、不定方向 |
| 运营官 Operator | Workbuddy | 不做判断、不做初建、不做调度 |
