# 变更评估（造价工程师版）

> 灵台"三步法"决策框架，已改造为新疆造价工程师场景。
> 适用范围：非提炼场景的变更/需求/效果分析决策入口。
> 提炼场景的原料决策见 [[10-索引与导航/索引与导航-总览]] 的 L1/L2/L3 分级确认制。

## 触发场景

| 场景 | 说明 | 入口 |
|------|------|------|
| 变更请求 | 业主/监理/上级提出修改/新增/优化需求 | 用户指令 |
| 效果分析 | 评估某条报价策略/签证方案/合同条款的执行效果 | 用户指令 |
| 工具采纳 | 评估是否引入新工具/新方法（如灵台脚本） | 用户指令 |

## 评估流程（三步法）

### 第1步：覆盖度判断

判断输入内容与现有知识的重叠程度：

| 结论 | 含义 | 下一步 |
|------|------|--------|
| **已有覆盖** | 核心内容已被知识库页面涵盖，无新概念 | 直接标记 已提炼，移入归档，结束 |
| **部分覆盖** | 核心框架已有，但含可落地的具体建议 | 提炼建议为变更项，提变更清单 |
| **全新内容** | 知识库没有对应页面或概念 | 提变更清单（增/改哪些页面） |

**判断标准**：核心设计理念是否已体现在知识库页面的标题/摘要/关键内容中？是 → 已有覆盖。

**造价场景示例**：
- 业主提出"按 GB 50500-2024 重新组价" → 已有覆盖（KC-107 已收录国标落地）
- 业主提出"按 GB 50500-2024 重新组价，且要求我们输出 ⚠ 风险点清单" → 部分覆盖
- 业主提出"按 GB 50500-2024 + 新疆补充子目动态调整文件" → 全新内容（如未入库）

### 第2步：变更影响评估

对"部分覆盖"或"全新内容"的情况，评估变更的规模：

| 维度 | 低影响 | 中影响 | 高影响 |
|------|:------:|:------:|:------:|
| 文件改动数 | ≤2 个 | 3-5 个 | >5 个 |
| 涉及技能文件 | 否 | 否 | 是 |
| 涉及规范重写 | 否 | 部分章节 | 是 |
| 影响现有工作流 | 不改变 | 补充细节 | 结构性修改 |

### 第3步：价值判断

| 因素 | 权重 | 问法 |
|------|:---:|------|
| 收益 | 高 | 做了之后对哪些造价场景有帮助？ |
| 成本 | 中 | 改动量多少？维护成本高吗？ |
| 风险 | 低 | 会不会破坏现有结构或流程？ |
| 时机 | 中 | 当前阶段值不值得做？ |

**造价特定风险点**：
- ⚠ 费率变更（管理费/利润/规费）→ 触发 GB 50500-2024 合规检查
- ⚠ 人工单价变更 → 触发信息价时效复核（90 天红线）
- ⚠ 增值税口径变更 → 区分一般纳税人 9% / 简易征收 3%
- ⚠ 合同条款变更 → 触发法律审查（GF-2017-0201 / FIDIC）

## 决策输出

评估完成后，落到以下四类输出之一：

```
输入 → 三步评估 → 决策输出
```

| 输出 | 标记 | 含义 | 后续操作 |
|------|:----:|------|----------|
| **直接做** | ✅ | 确定无风险，无需确认 | 老高直接执行，完成后写日志+提交 |
| **提变更清单** | 💬 | 有依赖用户判断的部分 | 列出清单 + 正反面分析，等昕田确认 |
| **暂缓** | ⏸ | 有价值但时机/规模不合适 | 记录在案，附原因 |
| **归档** | 🗄 | 已有覆盖或纯参考无提炼价值 | 标记 已处理，移入 15-数据资产/ |

### 决策优先级

1. **已有覆盖 → 优先归档**（不浪费时间在冗余判断上）
2. **部分覆盖 + 低影响 → 可考虑直接做**（但建议提变更让昕田知情）
3. **部分/全新 + 中高影响 → 必须提变更清单**
4. **效果分析 → 先出正反面分析，再交昕田决策**

> **经验法则：** AI 主动判断是否"已有覆盖"时，不必问用户。只有涉及"增/改/删"知识库页面时才必须提变更清单确认。
